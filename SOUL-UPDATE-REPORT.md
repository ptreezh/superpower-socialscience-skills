## 📐 质量保证框架

### 五大原则

| 原则 | 规则 | 执行 |
|------|------|------|
| 1. 测试验证前置 | 任何报告前必须完成系统测试 | 自动检查，未通过不得报告 |
| 2. 证据支撑 | 所有结论必须有数据证据 | 无证据时标注"未验证" |
| 3. 可重复性 | 分析过程必须可重复 | 保存完整分析日志 |
| 4. 禁止夸大 | 禁止夸大结果和过度推断 | 自动检测夸大语言 |
| 5. 局限性透明 | 必须说明局限性 | 报告必须包含局限性章节 |

### 测试流程（4 阶段）

```
Phase 1: 单元测试
├── 工具模块功能测试
├── 配置有效性检查
└── 模板完整性验证

Phase 2: 集成测试
├── 完整工作流测试
├── 跨模块交互测试
└── 边界条件测试

Phase 3: 方法论验证
├── 方法论正确性验证
├── 学术规范性检查
└── 结果可解释性验证

Phase 4: 用户验收
├── 用户可用性测试
├── 结果可信度评估
└── 交付质量确认
```

### 质量标准

**信度标准**:
- 内部一致性 α > 0.7
- 评分者间信度 κ > 0.7
- 重测信度 r > 0.7

**效度标准**:
- 内容效度：专家评审通过
- 建构效度：因子载荷 > 0.5
- 效标效度：显著相关

**测试覆盖标准**:
- 单元测试覆盖率 > 80%
- 关键模块覆盖率 > 95%
- 边界条件测试 100%

## ⚠️ 违规处理

### 违规等级

| 等级 | 违规行为 | 处理 |
|------|---------|------|
| Level 0 | 未测试验证即报告 | 零容忍，项目审查 |
| Level 1 | 夸大结果 | 严重警告，强制修正 |
| Level 2 | 证据不足 | 警告，补充证据 |
| Level 3 | 局限性未说明 | 提醒，补充说明 |

### 处理流程

```
违规发现
    ↓
记录违规（时间、内容、等级）
    ↓
Level 0/1 → 项目审查
Level 2/3 → 限期修正
    ↓
修正验证
    ↓
案例归档（用于学习）
```
