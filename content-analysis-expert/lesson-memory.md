## 成功模式

### 模式1：迭代编码方案开发

**场景**: 编码方案设计阶段

**方法**:
1. 文献回顾提取类目
2. 初步方案预测试
3. 根据问题修订
4. 再次测试验证
5. 正式方案确定

**效果**: 确保编码方案的可操作性和信度。

**适用条件**: 所有内容分析研究。

---

### 模式2：编码员培训协议

**场景**: 编码员准备阶段

**方法**:
1. 讲解研究背景和目的
2. 详细解释编码方案
3. 共同编码练习样本
4. 讨论分歧和边界案例
5. 独立编码测试样本
6. 检验一致性

**效果**: 提高编码员间一致性，减少后期问题。

**适用条件**: 需要多人编码的研究。

---

### 模式3：信度监控机制

**场景**: 正式编码阶段

**方法**:
- 定期抽取子样本双编码
- 计算累计信度
- 发现下降趋势及时干预
- 记录所有编码决策

**效果**: 确保编码质量稳定。

**适用条件**: 大规模编码项目。

## 失败教训

### 教训1：类目定义模糊

**场景**: 编码方案设计

**错误做法**:
```
类目: "正面报道"
定义: "态度积极的报道"
```

**后果**:
- 编码员理解不一致
- 信度无法达标
- 大量编码分歧

**正确做法**:
```
类目: "正面报道"
定义: "使用褒义形容词描述主体，或强调主体成就、进步的报道"
操作指标:
  - 褒义形容词数量 ≥ 2
  - 成就类动词使用
  - 无负面评价
编码示例: "该项目取得重大突破..."
```

**教训标签**: #定义模糊 #操作化不足

---

### 教训2：类目重叠

**场景**: 类目设计

**错误做法**:
```
类目A: "国内新闻" - 发生在国内的新闻
类目B: "社会新闻" - 涉及社会议题的新闻
→ 大量社会新闻发生在国内，同时归入两类
```

**后果**:
- 编码员困惑
- 编码结果不一致
- 数据解释困难

**正确做法**:
```
维度分离:
维度1 - 地域:
  - 国内
  - 国际
维度2 - 主题:
  - 政治
  - 经济
  - 社会
→ 每个维度独立编码，类目不重叠
```

**教训标签**: #类目重叠 #维度混淆

---

### 教训3：信度检验缺失

**场景**: 编码过程

**错误做法**:
- 单人编码不经检验
- 只计算简单百分比
- 信度低仍继续

**后果**:
- 数据可信度存疑
- 研究结论站不住脚
- 无法发表或被质疑

**正确做法**:
- 至少两名编码员独立编码
- 使用Krippendorff's α或Cohen's κ
- 信度低于0.80需调整方案

**教训标签**: #信度缺失 #方法不当

---

### 教训4：统计方法误用

**场景**: 数据分析

**错误做法**:
- 对频次数据使用t检验
- 忽视样本量对χ²的影响
- 过度解释相关关系

**后果**:
- 统计结论错误
- 结果无法复现
- 遭到审稿质疑

**正确做法**:
- 类目数据使用χ²检验
- 报告效应量
- 谨慎因果推断

**教训标签**: #统计误用 #过度推断

## 质量标准基准

### 信度标准

| 水平 | Krippendorff's α | Cohen's κ | 评价 |
|------|-----------------|-----------|------|
| 优秀 | ≥ 0.90 | ≥ 0.90 | 高质量 |
| 良好 | 0.80-0.89 | 0.80-0.89 | 可接受 |
| 最低 | 0.667-0.79 | 0.60-0.79 | 勉强 |
| 不可 | < 0.667 | < 0.60 | 需改进 |

### 编码质量指标

| 指标 | 标准 |
|------|------|
| 类目覆盖率 | 95%以上内容可归类 |
| 编码一致性 | 编码员间一致率 ≥ 85% |
| 编码速度 | 稳定后每小时单位数 |
