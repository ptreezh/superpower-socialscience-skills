# 设计思维方法论详细指南

## 概述

设计思维（Design Thinking）是一种以人为本的创新方法论，由IDEO和斯坦福d.school推广发展。它提供了一套系统化的方法来解决复杂问题，特别适用于用户需求不明确、解决方案不确定的创新场景。

## 理论基础

### 1. 人本设计哲学

设计思维的核心信念是：
- 用户是创新的起点和终点
- 深度同理心是洞察的来源
- 快速迭代优于完美规划
- 多元视角激发创新

### 2. 设计思维的认知基础

**发散思维与收敛思维**:
- 发散：生成大量可能性
- 收敛：筛选和聚焦最佳方案
- 设计思维在两者间交替

**系统性思考**:
- 问题不是孤立的
- 解决方案需要考虑系统影响
- 干预点的选择至关重要

## 五阶段详解

### 阶段1: 同理心洞察 (Empathize)

#### 目标
深入理解用户的真实需求、行为、痛点和情感。

#### 方法

**用户访谈**:
```
访谈类型:
├── 半结构化访谈
│   - 预设主题，灵活追问
│   - 平衡深度和广度
│
├── 深度访谈
│   - 聚焦核心体验
│   - 挖掘深层动机
│
└── 焦点小组
    - 群体互动
    - 观点碰撞
    - 共识识别
```

**用户观察**:
```
观察方法:
├── 上下文观察
│   - 在真实环境中观察
│   - 记录环境因素
│
├── 影子跟随
│   - 全程陪伴用户
│   - 记录完整旅程
│
└── 行为痕迹分析
    - 分析使用痕迹
    - 推断行为模式
```

**同理心地图**:
```
         ┌──────────────────┐
         │    想 (Think)    │
         │  内心想法和信念   │
         └────────┬─────────┘
                  │
┌─────────┐       │       ┌─────────┐
│ 说 (Say)│───────┼───────│ 做 (Do) │
│用户原话 │       │       │ 具体行为 │
└─────────┘       │       └─────────┘
                  │
         ┌────────▼─────────┐
         │    感 (Feel)     │
         │    情感和情绪     │
         └──────────────────┘
```

#### 输出物
- 用户画像（User Personas）
- 同理心地图（Empathy Maps）
- 用户旅程图（User Journey Maps）
- 关键洞察清单（Key Insights）

### 阶段2: 问题定义 (Define)

#### 目标
将分散的用户洞察整合为清晰的设计挑战。

#### 方法

**POV陈述**:
```
POV = Point of View

结构: [用户]需要[需求]，因为[洞察]

示例:
"忙碌的职场妈妈需要高效的购物解决方案，
因为她们在工作和家庭之间缺乏时间进行日常采购。"
```

**HMW问题重构**:
```
HMW = How Might We

转换规则:
- 从问题陈述中提取核心矛盾
- 转化为"我们如何能够..."的问题
- 问题要足够宽泛，允许创新空间
- 问题要足够聚焦，具有可操作性

示例:
问题: "妈妈们没有时间购物"
HMW: "我们如何能够让购物融入妈妈们的碎片时间？"
```

**问题空间探索**:
```
问题空间:
├── 表面问题（症状层）
├── 深层问题（原因层）
├── 用户视角问题
├── 利益相关者视角问题
└── 系统视角问题
```

#### 输出物
- POV陈述
- HMW问题列表
- 设计挑战定义
- 问题空间地图

### 阶段3: 创意发散 (Ideate)

#### 目标
生成大量创新方案，突破常规思维。

#### 方法

**头脑风暴规则**:
```
1. 不评判 - 先发散，后收敛
2. 数量优先 - 想法越多越好
3. 在他人想法上构建 - "是的，而且..."
4. 鼓励疯狂想法 - 疯狂中往往有突破
```

**SCAMPER创新法**:
```
S - Substitute (替代)
    能否用其他材料/方法/人员替代？

C - Combine (组合)
    能否与其他功能/产品组合？

A - Adapt (适应)
    能否借鉴其他行业的做法？

M - Modify (修改)
    能否改变形状/颜色/声音？

P - Put to other use (其他用途)
    能否用于其他场景或用户？

E - Eliminate (消除)
    能否去掉不必要的部分？

R - Reverse (反转)
    能否颠倒顺序或逻辑？
```

**类比创新**:
```
类比类型:
├── 自然类比
│   - 向自然学习设计
│   - 仿生学应用
│
├── 跨界类比
│   - 借鉴其他行业
│   - 移植成功模式
│
└── 极端用户类比
    - 从极端用户学习
    - 发现隐藏需求
```

#### 输出物
- 创意概念库（100+想法）
- 概念聚类图
- 优选方案列表（3-5个）
- 概念评估矩阵

### 阶段4: 原型制作 (Prototype)

#### 目标
用最小成本验证核心假设。

#### 方法

**假设优先级矩阵**:
```
           高风险
             │
    ┌────────┼────────┐
    │   优先  │   重要  │
    │  原型   │  原型   │
    ├────────┼────────┤
    │   可选  │   延后  │
    │  原型   │  原型   │
    └────────┼────────┘
             │
           低风险
      不确定 ←───→ 确定
```

**原型类型选择**:
```
原型类型:
├── 故事板原型
│   - 成本: 低
│   - 保真度: 低
│   - 适用: 概念验证
│
├── 纸面原型
│   - 成本: 低
│   - 保真度: 低
│   - 适用: 交互流程验证
│
├── 角色扮演
│   - 成本: 低
│   - 保真度: 中
│   - 适用: 服务流程验证
│
├── 数字线框
│   - 成本: 中
│   - 保真度: 中
│   - 适用: 界面设计验证
│
└── 功能原型
    - 成本: 高
    - 保真度: 高
    - 适用: 可用性测试
```

#### 输出物
- 假设清单
- 原型设计文档
- 测试脚本
- 测试材料

### 阶段5: 用户测试 (Test)

#### 目标
收集真实用户反馈，指导迭代方向。

#### 方法

**可用性测试**:
```
测试流程:
├── 准备阶段
│   - 定义测试目标
│   - 选择测试用户
│   - 准备测试任务
│   - 准备记录工具
│
├── 执行阶段
│   - 引导用户操作
│   - 观察用户行为
│   - 记录用户反馈
│   - 注意情绪变化
│
└── 分析阶段
    - 整理测试记录
    - 识别问题模式
    - 提取改进建议
    - 制定迭代计划
```

**反馈分析方法**:
```
反馈类型:
├── 行为反馈（用户做了什么）
├── 语言反馈（用户说了什么）
├── 情绪反馈（用户感受如何）
└── 建议 feedback（用户建议什么）

分析框架:
├── 严重问题（阻碍任务完成）
├── 一般问题（影响体验）
├── 小问题（可以优化）
└── 正面反馈（保持优点）
```

#### 输出物
- 测试报告
- 问题清单
- 用户反馈汇总
- 迭代建议

## 迭代循环

设计思维不是线性流程，而是迭代循环：

```
Empathize → Define → Ideate → Prototype → Test
    ↑                                       │
    └───────────────────────────────────────┘
```

**迭代触发条件**:
- 核心假设被证伪
- 用户反馈负面超过50%
- 发现新的用户需求
- 技术可行性问题
- 商业可行性问题

## 质量标准

### 同理心阶段质量标准
- [ ] 至少访谈5-8个用户
- [ ] 用户画像完整详细
- [ ] 同理心地图四象限完整
- [ ] 关键洞察有证据支撑

### 定义阶段质量标准
- [ ] POV陈述清晰有力
- [ ] HMW问题开放聚焦平衡
- [ ] 问题定义有洞察深度
- [ ] 利益相关者认可

### 创意阶段质量标准
- [ ] 生成至少100个想法
- [ ] 使用多种发散方法
- [ ] 筛选标准明确
- [ ] 方案多样化

### 原型阶段质量标准
- [ ] 假设优先级清晰
- [ ] 原型类型合适
- [ ] 制作成本低
- [ ] 可测试性强

### 测试阶段质量标准
- [ ] 测试用户代表目标群体
- [ ] 测试任务真实
- [ ] 反馈收集系统
- [ ] 迭代方向明确

## 常见问题

### Q1: 如何判断是否需要回到同理心阶段？
A: 当用户反馈与设计假设严重不符，或发现新的用户群体时，应回到同理心阶段。

### Q2: 创意数量多少才够？
A: 建议每30分钟头脑风暴生成至少20个想法，总体目标100+。

### Q3: 原型应该多精细？
A: 原型精度取决于测试目标。验证概念用低保真，验证可用性用高保真。

### Q4: 测试需要多少用户？
A: 定性测试5-8个用户通常能发现80%的可用性问题。
