# DID长时研究项目指南

## 场景1: 医院信息系统实施研究（3-6个月）

### Phase 1: 初始情境分析（Month 1）

#### Week 1-2: 利益相关者深度映射
**Task**: 识别并分析所有利益相关者

**主要利益相关者**:
1. **临床医生**
   - 权力: 中等（专业权威）
   - 利益: 高（直接影响工作）
   - 主要关切: 工作量、专业自主性、患者关系

2. **护士人员**
   - 权力: 低-中等
   - 利益: 高
   - 主要关切: 文书负担、沟通效率

3. **医院管理层**
   - 权力: 高（决策权）
   - 利益: 高（绩效、效率）
   - 主要关切: 效率、质量控制、成本

4. **信息技术部门**
   - 权力: 中等（技术专长）
   - 利益: 中等
   - 主要关切: 系统稳定性、技术支持

5. **患者**
   - 权力: 低（但重要）
   - 利益: 高（医疗服务质量）
   - 主要关切: 隐私、服务质量、等待时间

**输出**:
- 利益相关者权力-利益矩阵
- 利益冲突分析报告
- 主要联盟和反对力量识别

#### Week 3-4: 技术要素详细清单
**Task**: 打开技术黑箱，识别所有技术要素

**技术要素清单**:
1. **电子病历系统 (EMR)**
   - 脚本: 标准化流程
   - 约束: 必填项、模板
   - 能力: 数据记录、检索

2. **计算机化医嘱录入系统 (CPOE)**
   - 脚本: 规范化处方
   - 约束: 药物相互作用检查
   - 能力: 决策支持

3. **临床决策支持系统 (CDSS)**
   - 脚本: 最佳实践推荐
   - 约束: 警告、提醒
   - 能力: 知识管理

4. **移动设备终端**
   - 脚本: 随时随地访问
   - 约束: 屏幕尺寸、电池
   - 能力: 移动性

5. **数据分析和报告工具**
   - 脚本: 绩效监控
   - 约束: 数据质量要求
   - 能力: 管理控制

**输出**:
- 技术要素详细清单
- 每个要素的脚本分析
- 技术脚本与实践的冲突点识别

### Phase 2: 迭代周期分析（Months 2-5）

#### 迭代周期1: 系统引入（Month 2）

**技术迭代**:
- **设计阶段**: 系统选型、定制化需求
- **实施阶段**: 部署、培训、试运行
- **反思阶段**: 问题识别、初步调整

**社会迭代**:
- **协商阶段**: 不同科室的需求表达
- **适应阶段**: 医生的初始使用策略
- **重构阶段**: 工作流程的初步调整

**耦合点分析**:
1. **标准化 vs 个性化**: 系统模板 vs 医生个性化需求
2. **效率 vs 质量**: 快速录入 vs 详细记录
3. **监控 vs 信任**: 管理监控 vs 专业自主

**输出**:
- 周期1完整报告
- 关键耦合点识别
- 初步机制假设

#### 迭代周期2: 调整与抵抗（Months 3-4）

**技术迭代**:
- **设计调整**: 基于反馈修改界面、流程
- **实施调整**: 增加培训、技术支持
- **反思调整**: 识别持续使用问题

**社会迭代**:
- **协商深化**: 不同科室的差异应对
- **适应策略**:
  - 绕过: 复制粘贴、事后补录
  - 反抗: 拒绝使用、抱怨
  - 改造: 创造性使用模板
  - 服从: 严格按系统要求

**耦合点深化**:
1. **适应策略触发机制**: 为什么医生选择绕过?
2. **组织激励影响**: 绩效考核如何影响使用?
3. **专业文化重塑**: 系统如何改变医疗文化?

**输出**:
- 周期2完整报告
- 适应策略类型学
- 机制提炼

#### 迭代周期3: 稳定化与再谈判（Month 5）

**技术迭代**:
- **功能固化**: 成功的修改成为标准
- **功能扩展**: 新功能需求识别
- **长期维护**: 系统优化计划

**社会迭代**:
- **制度化**: 新的工作流程成为常规
- **角色重构**: IT部门权力上升
- **规范变迁**: 新的专业标准形成

**耦合点稳定**:
1. **新的平衡**: 技术-社会新的稳定状态
2. **权力重组**: 谁的权力增加/减少?
3. **意义重构**: 系统被赋予什么意义?

**输出**:
- 周期3完整报告
- 演化轨迹总结
- 稳定机制分析

### Phase 3: 中层理论生成（Month 6）

#### Task 3.1: 机制命题提炼

**从案例到机制**:
```
观察: 医生修改病历频率增加
  ↓
机制: 技术脚本刚性 + 绩效考核压力 → 适应策略
  ↓
命题:
"当技术脚本的刚性与组织绩效考核结合时，
会触发用户的适应性策略（如复制粘贴），
导致技术目标（规范化）被 undermined。"
```

**其他机制**:
1. **专业自主性侵蚀机制**
2. **工作负载转移机制**
3. **知识固化机制**

**输出**:
- 中层理论命题集（至少3个）
- 每个命题的逻辑链
- 边界条件讨论

#### Task 3.2: 跨案例验证

**验证情境**:
1. **其他医院**: 类似系统、不同组织文化
2. **其他领域**: 教育、政务系统
3. **不同国家**: 制度环境差异

**边界条件**:
- 适用于强监控环境
- 适用于标准化要求高的职业
- 适用于专业自主性强的群体

**输出**:
- 跨案例比较报告
- 命题修正和精炼
- 最终理论报告

## 场景2: 教育平台数字化转型研究（4-8个月）

### Phase 1: 初始情境分析（Months 1-2）

**利益相关者**:
- 教师（专业自主性、教学风格）
- 学生（学习体验、隐私）
- 学校管理（质量监控、标准化）
- 平台提供者（商业模式、用户增长）
- 家长（教育焦虑、参与感）

**技术要素**:
- 学习管理系统 (LMS)
- 在线测试工具
- 视频会议系统
- 学习分析仪表板
- 通信工具

### Phase 2: 迭代周期分析（Months 3-6）

#### 迭代周期1: 疫情应急（Month 3）
- 技术迭代: 快速部署、功能简化
- 社会迭代: 强制使用、混乱适应
- 耦合点: 技术能力 vs 教学需求

#### 迭代周期2: 混合教学（Months 4-5）
- 技术迭代: 功能优化、整合
- 社会迭代: 教学策略创新、疲劳
- 耦合点: 线上 vs 线下教学平衡

#### 迭代周期3: 新常态（Month 6）
- 技术迭代: 功能固化、新功能
- 社会迭代: 制度化、新教学文化
- 耦合点: 教育重塑

### Phase 3: 中层理论生成（Months 7-8）

**关键机制**:
1. **教学技术化机制**: 技术重塑教学实践
2. **学习监控化机制**: 数据重塑学习体验
3. **教育商品化机制**: 平台商业模式重塑教育

**输出**:
- 中层理论命题
- 跨情境验证（K-12 vs 高等教育）
- 政策建议

## 场景3: 数字政务系统参与式设计研究（6-12个月）

### Phase 1: 初始情境分析（Months 1-2）

**利益相关者**:
- 政府官员（效率、透明度、政绩）
- 公民（便利性、隐私、参与）
- 技术开发商（合同、技术实现）
- 非政府组织（监督、倡导）
- 学术研究者（评估、批判）

**技术要素**:
- 在线服务平台
- 数据共享系统
- 身份认证系统
- 反馈和投诉系统
- 开放数据门户

### Phase 2: 迭代周期分析（Months 3-10）

#### 迭代周期1: 自上而下设计（Months 3-4）
- 技术迭代: 政府主导设计
- 社会迭代: 公民使用、反馈
- 耦合点: 官僚逻辑 vs 用户需求

#### 迭代周期2: 参与式调整（Months 5-7）
- 技术迭代: 基于反馈修改
- 社会迭代: 公民参与、抗议
- 耦合点: 透明度 vs 隐私

#### 迭代周期3: 制度化与挑战（Months 8-10）
- 技术迭代: 系统稳定化
- 社会迭代: 数字鸿沟、信任问题
- 耦合点: 效率 vs 公平

### Phase 3: 中层理论生成（Months 11-12）

**关键机制**:
1. **数字化官僚主义机制**: 技术强化官僚控制
2. **数字参与机制**: 技术创造新的参与形式
3. **数字信任机制**: 技术影响政府-公民信任

**输出**:
- 中层理论命题
- 跨国比较（不同制度环境）
- 治理启示

## 跨对话状态保持

### 状态文件结构

```
project-root/
├── task_plan.md              # 总体研究计划
├── progress.md               # 进度日志
├── stakeholders.md           # 利益相关者分析
├── technical_elements.md     # 技术要素清单
├── iteration_1/              # 迭代周期1
│   ├── technical.md
│   ├── social.md
│   └── coupling.md
├── iteration_2/              # 迭代周期2
│   ├── technical.md
│   ├── social.md
│   └── coupling.md
├── mechanisms.md             # 机制识别
├── propositions.md           # 中层理论命题
└── final_report.md           # 最终报告
```

### 会话恢复协议

**每次会话开始**:
1. 读取 `task_plan.md` 和 `progress.md`
2. 识别当前阶段和未完成任务
3. 加载相关数据和发现

**每次会话结束**:
1. 更新 `progress.md`（新增发现、问题）
2. 创建 checkpoint（保存关键数据）
3. 规划下次会话任务

**checkpoint 策略**:
- 每完成一个 Phase: 创建完整 checkpoint
- 每完成一个迭代周期: 保存周期数据
- 每识别一个机制: 保存机制分析
