### 🌟 子Agent并行支持（增强）

**何时使用子Agent并行**:

SNA分析特别适合并行处理的场景：
- ✅ 多个时间点的网络演化分析（时间序列SNA）
- ✅ 多个独立网络的比较分析
- ✅ 大规模网络的分块计算
- ✅ 批量计算多种中心性指标

**决策逻辑**:

```python
def decide_sna_mode(analysis_tasks):
    """SNA分析决策逻辑"""

    # 场景1: 单个网络分析
    if len(analysis_tasks) == 1:
        return "CLI_QUEUE", "单个网络，使用CLI队列"

    # 场景2: 多个时间点（时序SNA）
    if is_temporal_analysis(analysis_tasks):
        if len(analysis_tasks) > 5:
            return "SUBAGENT_PARALLEL", "时间点多，使用并行"
        else:
            return "CLI_QUEUE", "时间点少，使用CLI队列"

    # 场景3: 多个独立网络
    if are_independent_networks(analysis_tasks):
        if len(analysis_tasks) > 5:
            return "SUBAGENT_PARALLEL", "网络多，使用并行"
        else:
            return "CLI_QUEUE", "网络少，使用CLI队列"

    # 默认：CLI队列
    return "CLI_QUEUE", "标准SNA分析"
```

**时间序列SNA示例**:

```yaml
场景: 分析20个时间点的合作网络演化

CLI队列（串行）:
  时间点1-20: 依次分析
  每个时间点: 15分钟
  总时间: 20 × 15 = 300分钟（5小时）

子Agent并行（批量）:
  启动20个sna-analyzer子Agent
  所有时间点同时分析
  总时间: max(15分钟) = 20分钟
  加速比: 15x ⚡

实现:
  for time_point in range(1, 21):
      subagent = Agent(
          subagent_type="sna-analyzer",
          prompt=f"分析时间点{time_point}的网络",
          run_in_background=True
      )
```

**优雅降级**:

```yaml
保证: 即使子Agent不可用，SNA分析仍正常完成

降级场景:
  1. Agent tool不可用
     → 降级到CLI队列
     → 依次分析每个时间点

  2. 部分子Agent失败
     → 失败的时间点降级到CLI队列
     → 成功的保留结果

  3. 资源不足
     → 分批处理（每批5个时间点）
     → 保证所有时间点都能分析
```

**用户控制**:

## 🔧 工作流程（嵌入绝对禁止检查）

```
用户提供网络数据（边列表、邻接矩阵等）
    ↓
【前置检查】任务分解质量验证
  - 粒度是否足够在一次会话中完成？
  - 是否有明确的量化标准？
  - 是否有完成度检查清单？
  - 是否要求子 agent 输出数据清单？
    ↓
1. 数据准备阶段
   - 数据格式验证
   - 数据清洗（孤立点、自环）
   - 网络构建
   - 基础指标计算（密度、直径、聚类系数）
   - 【检查点】网络构建是否完整？所有节点是否包含？
    ↓
2. 中心性分析阶段
   - 度中心性计算与解读
   - 介数中心性计算与解读
   - 接近中心性计算与解读
   - 特征向量中心性计算与解读
   - 【检查点】所有指标是否覆盖所有节点？是否有解读？
   - 【禁止】只计算部分中心性指标？前详后略？
    ↓
3. 社群检测阶段
   - Louvain 算法社群发现
   - 标签传播算法验证
   - 社群质量评估（模块度）
   - 社群特征分析
   - 【检查点】社群检测是否完整？模块度是否合理？
   - 【禁止】只运行一种算法？忽略孤立节点？
    ↓
4. 结构洞分析阶段
   - 约束指标计算
   - 有效规模计算
   - 桥接角色识别
   - 结构洞机会分析
   - 【检查点】结构洞指标是否完整？识别是否准确？
   - 【禁止】只计算部分指标？忽略负案例？
    ↓
5. 可视化阶段
   - 整体网络布局（力导向）
   - 中心性可视化（节点大小/颜色）
   - 社群结构可视化
   - 交互式探索
   - 【检查点】可视化是否清晰？标签是否可读？
   - 【禁止】节点过载？颜色混淆？
    ↓
6. 报告生成阶段
   - 整合所有分析结果
   - 撰写方法论说明
   - 提供结果解读
   - 提出策略建议
   - 【检查点】报告是否包含所有部分？解读是否充分？
   - 【禁止】结果空洞？只有图表没有解读？
    ↓
【最终检查】完成度验证清单（15 项）
    ↓
【承诺宣读】确认未违反任何禁止原则
    ↓
报告完成（仅当全部验证通过）
```

## 📋 任务分解规则

### 分解原则

1. **粒度可控**：每个子任务必须能在一次会话中完成
2. **量化标准**：每个子任务必须有明确的完成标准
3. **独立验证**：每个子任务完成后必须独立验证
4. **数据清单**：子 agent 必须输出完整的数据清单

### 分解模板

```
总任务：[任务名称]
  ├─ 子任务 1: [具体描述]
  │   - 完成标准：[量化指标]
  │   - 输出清单：[文件列表]
  │   - 验证方法：[检查步骤]
  │
  ├─ 子任务 2: [具体描述]
  │   - 完成标准：[量化指标]
  │   - 输出清单：[文件列表]
  │   - 验证方法：[检查步骤]
  │
  └─ ...
```

### 委托要求

**必须明确**：
- 任务范围（具体节点/指标/算法）
- 完成标准（覆盖率、指标类型、解读要求）
- 输出清单（文件名、路径、内容要求）
- 验证方法（检查步骤、合格标准）

**禁止模糊**：
- "分析一些节点"
- "计算几个指标"
- "大概看看就行"
- "看着办"

## 🎓 质量标准

### 分析质量评估矩阵

| 维度 | 合格标准 | 检查方法 |
|------|---------|---------|
| **数据完整性** | 所有节点和边都被包含 | 计数验证 |
| **中心性分析** | 4项指标全部计算并解读 | 指标清单检查 |
| **社群检测** | 至少2种算法，有模块度评估 | 报告检查 |
| **结构洞分析** | 约束指标、有效规模、桥接角色完整 | 结果检查 |
| **可视化质量** | 布局清晰，标签可读，颜色有意义 | 视觉检查 |
| **报告完整性** | 方法论、结果、解读、建议四部分齐全 | 报告检查 |
| **禁止原则遵守** | 完全遵守所有禁止原则 | 逐项检查 |

### 违规后果

| 违规类型 | 后果 | 纠正措施 |
|---------|------|---------|
| 懒惰走捷径 | 网络特征遗漏，分析不完整 | 重新完成所有分析 |
| 未验证就报告 | 研究质量无法保证 | 逐项验证后重新报告 |
| 追求完成感 | 网络洞察存疑 | 如实报告真实进度 |
| 标准不一致 | 数据可信度下降 | 统一标准后重新分析 |
| 掩盖实质缺失 | 浪费用户时间 | 公开承认并补充完成 |
| 牺牲分析质量 | 学术声誉受损 | 接受批评并重新分析 |
